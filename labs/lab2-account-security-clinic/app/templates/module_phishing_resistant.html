What “phishing-resistant” means

A method is phishing-resistant when it is difficult to trick a user into authenticating to a fake site and handing an attacker reusable secrets. Examples include security keys and passkeys (WebAuthn).

High-level takeaway: Prefer authentication that is bound to the legitimate site and device, not just “a code the user can type anywhere.”

Compare common methods

SMS codes: easy to deploy, higher risk (SIM swap, phishing)
Authenticator OTP: better than SMS, still phishable if user types into fake site
Push approve/deny: user-friendly, vulnerable to push fatigue and social engineering
Number matching: improved push; reduces blind approval
Passkeys/WebAuthn: phishing-resistant; device-bound; improving usability
Security keys: strong phishing resistance; great for admins/high-risk roles

Rollout tips (schools):
1) Start with staff/admin accounts and high-impact systems (email, SIS, Google/M365 admin).
2) Use passkeys where possible; keep a safe recovery process.
3) Teach “never type codes into links from messages” and promote bookmarked sign-in URLs.
4) For shared labs: avoid shared accounts; use role-based access and auditing.

Teaching prompts

Prompt 1: What’s the biggest barrier to passkeys/security keys in your environment: cost, device compatibility, training, or support?
Prompt 2: Which accounts should get the strongest MFA first? Build a priority list and justify it.
Prompt 3: Define one policy change you could implement this month that meaningfully reduces phishing risk.
Simple assessment idea: Ask learners to classify MFA methods into: - Easily phishable - Less phishable - Phishing-resistant Then explain one tradeoff for each.
